# 工作指南

- 使用中文交流。
- 永远不要执行 `git clean`（任何参数，尤其是 `-fdx`）。
- `No Spec, No Code`
- `No Approval, No Execute`
- `Spec is Truth`

## 此项目重点

1. 核心架构是 agent loop + skill，不要作多余的事情。
2. 不要修改 `agent_service.py`，如果必须修改，需要提前告知我原因。
3. 保证整体架构的干净整洁，不修改架构设计，不做冗余代码。

## Skill 使用约定

- 默认使用 `sdd-riper-one-light`
- 复杂任务使用：`sdd-riper-one`
  复杂任务包括但不限于：
  - 架构设计或大范围重构
  - 跨模块 / 跨项目改动
  - 需求模糊，需要多方案权衡
  - 长链路调试、复杂迁移、严格评审流程
- 极简单、无需 spec / checkpoint / 分阶段控制的任务，可以不使用 skill。

## 执行规则

- 代码修改前先提交方案并等待我确认；文档修改可直接执行。
- 等待确认期间，先将方案收敛到最小改动集，不要无限扩展搜索范围。
- 需要创建新文件/新文档的时候，先创建空文件，填充标题，然后分段填充剩余内容。
- 修改文件时，优先小步、分段、少量多次修改，防止上下文爆炸。
- 除非我明确要求，否则不要做大范围重构、整文件替换或超大 patch。
- 多文件改动时，先完成最核心链路的最小可用修改，再逐步补齐。
- 不要顺手修改与当前任务无关的内容。
- 如果连续 2 轮以上仅阅读/搜索而没有实际修改，先停止继续探索，并明确说明准备修改哪些文件、先改哪一处、为什么。

## 命令边界

- 除非我明确要求，否则不要主动运行编译、打包、测试、部署、迁移或其他高开销命令。
- 除非我明确要求，否则不要主动安装依赖、升级依赖、删除依赖或修改锁文件。
- 对可能造成不可逆后果、批量删除、覆盖、重置、迁移或外部写操作的命令，先等待我确认。

## Git 边界

- 默认严格尊重 `.gitignore` 与所有已忽略路径。
- 不要主动使用 `git add -f` / `git add --force`。
- 被 `.gitignore` 忽略的内容默认不提交；只有我明确点名路径时，才允许精确提交。
- 使用中文编写 Git commit，要求客观、精确。必须清晰地说明本次提交具体修改了什么内容以及修改的原因，确保团队成员单看 commit message 就能完全掌握改动细节。

## Spec 同步

- 当改动影响需求、接口、行为、约束、流程或实现决策时，执行后同步更新 spec。
- 纯机械性改动可不更新 spec。
